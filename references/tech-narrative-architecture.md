# 技术叙事架构

方法源自 [GitHub awesome-copilot / repo-story-time](https://github.com/github/awesome-copilot/tree/main/skills/repo-story-time)，将零散技术点转化为「背景 → 问题 → 演进 → 方案」完整叙事线。

---

## 1. repo-story-time 方法论拆解

repo-story-time 的核心能力：

| 阶段 | 原方法 | 技术写作映射 |
|------|--------|-------------|
| Phase 1 地形勘探 | 分析仓库文件结构、配置文件、README | 分析技术生态：竞品、替代方案、历史背景 |
| Phase 2 技术深潜 | 归纳架构、组件、数据流、技术栈 | 归纳核心机制、数据结构、设计哲学 |
| Phase 3 历史考古 | 提取提交统计、贡献者、活动模式、变更模式 | 提取版本演进、关键论文、里程碑事件、设计决策 |
| Phase 4 模式识别 | 识别角色、季节、主题、冲突、演化 | 识别技术冲突、设计取舍、范式转移 |

核心洞见：**技术文章不能只交付"方案"（Solution），必须交付完整的叙事弧线——背景张力、问题驱动、演进过程，最后才是方案理解。**

---

## 2. 叙事线四要素

### 2.1 背景 — Context

> 不是堆叠时间线，而是制造"当时局面的张力"。

**三问法**：
- 当时的技术环境是什么状态？（硬件/软件/成本/性能约束）
- 当时的主流方案有什么隐含缺陷？（不是出问题，而是"将就"）
- 什么外部变化打破了既有平衡？（硬件突破 / 场景爆发 / 理论突破）

**写作纪律**：
- ❌ "随着计算机技术的发展，内存容量越来越大……"
- ✅ "1980 年代，内存 64KB 起步，给每个进程分配固定地址空间还能凑合——但到了 1990 年代初，多任务操作系统普及，物理地址直接暴露给进程的设计再也撑不住了。"
- 关键：**具体年代、具体容量、具体场景、具体矛盾点**

### 2.2 问题 — Problem

> 不是"这个技术解决了什么问题"，而是"当时的人面临什么具体的痛苦"。

**问题包装三原则**：
1. **场景化**：放在一个具体开发/使用场景中（"你在写 OS 内核时……"）
2. **代价化**：不解决会付出什么具体代价（"一个 buggy 进程能把整个系统写挂，数据全丢"）
3. **两难化**：当时的人面临的真实取舍（"用 MMU 太慢，不用又不安全"）

**检查标准**：读者读到问题描述时，应该产生"这确实是个问题，那后来呢？"的悬念。

### 2.3 演进 — Evolution

> 不是罗列版本号，而是展示"人类如何一步步逼近答案"。

**叙事手法**：
- **试错法**：早期方案 → 暴露什么缺陷 → 怎么改进 → 又有新的问题
- **关键突破法**：某个人的某个洞察改变了方向（"直到一个人提出……"）
- **外在驱动法**：不是技术自己在发展，是外部条件变了（"当 CPU 从单核变成多核，原来认为正确的东西全部需要重想"）

**写作纪律**：
- 每个演进阶段都要标注当时的约束条件（硬件、成本、知识水平）
- 避免"事后诸葛亮"心态——第一阶段的设计在那个年代是合理的
- 演进动力必须有具体的外因，不写"随着技术发展"

### 2.4 方案 — Solution

> 不是罗列功能列表，而是解释"为什么最终长成了这个形状"。

**方案交付标准**：
1. 它最终长什么样？（架构 / 数据结构 / 关键路径）
2. 它做了哪些取舍？（选择这个放弃那个，为什么）
3. 它还有什么没解决？（开放问题 / 局限性 / 未来方向）

## 3. 三种叙事原型

### 原型 A：演进史式（适合框架一：原理演进型）

结构：「背景出现矛盾 → 旧方案失效 → 新方案诞生 → 新方案带来新问题 → 继续演进」

```
张力建立（旧方案能做什么，不能做什么）
   ↓ 矛盾积累
 第一次突破（新方案解决了一个问题，但制造了另一个）
   ↓ 新的不足
 第二次突破（更优的取舍找到了）
   ↓ 现状
 今天的方案 + 遗留的开放问题
```

**适用**：OS 调度器演进、文件系统发展史、内存管理演化

**对应七段式章节映射**：
- 段一（受众定位）→ 不单独出现，融入背景
- 段二（问题定义）→ 张力建立 + 矛盾积累
- 段三-四（概念+原理）→ 突破 1 → 突破 2
- 段五（实践）→ 今天的方案
- 段六（坑点）→ 开放问题 + 取舍

### 原型 B：对抗式（适合框架二：技术对比型 / 揭秘型）

结构：「一个公认做法 / 一个看似完美的方案 → 被发现有一个致命问题 → 另一条路线胜出」

```
权威建立（"当时所有人都认为 X 是对的"）
   ↓ 异常出现（"但有一个团队发现，X 在某些条件下并不成立"）
 对抗展开（旧派的反驳、证据的积累）
   ↓ 范式转移（新解释模型胜出）
 现状（当今天怎么看这个"对抗"）
```

**适用**：eBPF vs 内核模块、RISC vs CISC、GC vs 手动管理

**关键**：对抗式不是为了制造 drama，而是展示技术决策的复杂性。

**子类型 B1 — 均衡对比型**（适合框架二：技术对比型，当双方无明确胜负时）：

当对比的两个技术方案不存在"一方取代另一方"的范式转移，而是**各自解决了不同场景的问题、各有取舍**时，对抗式的"一方胜出"骨架会扭曲技术判断。此时使用均衡对比型：

```
引入（两边同时摆出——"做同一件事，两种路线"）
   ↓ 共同情境（"什么场景下，这两个方案同时出现了？"）
  双方案并置（A 的设计选择 + 取舍代价 ←→ B 的设计选择 + 取舍代价）
   ↓ 对比表（多维度：每维都标注"哪方在哪端更强"，而非"哪方赢了"）
  选型分界（"当条件为 X 时选 A，当条件为 Y 时选 B"——给出可验证的分界条件）
   ↓ 共存前景（"它们会互相取代吗？大概率不会，因为底层假设不同"）
```

结构说明：

| 段 | 叙事角色 | 纪律 |
|----|---------|------|
| 一 | 引入共同问题 | 展示两个方案要解决的**同一个具体痛点**，让读者理解"为什么同时需要两个" |
| 二 | A 方案深拆 | 设计哲学 + 核心机制 + 做了哪些取舍（不美化、不贬低） |
| 三 | B 方案深拆 | 同上结构，用词一致，保持分析对称 |
| 四-A | 对比表 | 多维度分列，每个维度标注**分界条件**（"低于 X 选 A，高于 X 选 B"） |
| 四-B | 代码对比 | 同一任务两种实现，读者可对照差异 |
| 五 | 选型建议 | 给出 3-5 个具体场景 + 推荐 + 理由 |
| 六 | 开放问题 | 两边都没解决的共同空白、未来的收敛可能性 |

**关键纪律**：
- 禁止"一方更优"的语言框架（"完胜""碾压""替代"）
- 对比表的每个维度必须对应**具体取舍**（不是"A 快，B 慢"，而是"A 在冷启动快但运行时慢，B 相反"）
- 选型建议必须给出**可验证的分界条件**（"工具数 ≤ 5 选 A，≥ 15 选 B"）而非笼统的"看场景"
- 如果两边的底层假设不同（如进程模型 vs API 增强），在对比表中另开一行标注，不做"苹果对苹果"的强比较
- 允许结论是"两者长期共存"，但必须给出各自的最佳生态位

### 原型 C：揭秘式（适合框架三：实践指南型 / 框架四：源码拆解型）

结构：「你看到的表面现象 → 背后的真实机制 → 设计者的真实意图」

```
表象（"这是 Linux 调度器——看起来它公平地分配 CPU 时间"）
   ↓ 追问（"但谁来决定什么是公平？偏重 IO 还是偏重计算？"）
 深层机制（"原来背后有一套复杂的权重系统和补偿机制"）
   ↓ 设计意图还原（"这么设计是因为……不是简单的公平，而是……"）
 你的视角切换（"看完这个，你再看到 htop 时，看到的就不一样了"）
```

**适用**：内核源码拆解、算法实现分析、系统调用路径追踪

## 4. 从散点到叙事的转换 SOP

### Step 1：信息采集（对应 repo-story-time Phase 1-2）

在开始写作前，先采集以下信息：

```markdown
### 背景信息表
- 技术诞生时间：____
- 当时的主流硬件/软件环境：____
- 当时的主要竞品/替代方案：____
- 催生该技术的外部推力：____（硬件突破/需求爆发/理论发现）

### 核心机制表
- 该技术解决的问题：____
- 解决问题的核心机制：____
- 关键数据结构/算法：____
- 做了哪些取舍：____

### 演进信息表
- 第一个版本/论文/实现时间：____
- 重大转折节点（至少 2 个）：____
- 每次转折的外部原因：____
- 关键人物/团队贡献：____
```

### Step 2：技术考古（对应 repo-story-time Phase 3）

找到以下信息的**具体值**，而非模糊表述：

| 原写法 | 写具体 |
|--------|--------|
| "早期计算机内存很小" | "1980 年代内存 64KB-256KB，一个进程占 16KB，最多跑 4 个进程" |
| "后来硬件升级了" | "2005 年 Intel 引入双核处理器，内存带宽从 6.4GB/s 升至 12.8GB/s" |
| "X 比 Y 快" | "在 CPU 4GHz 时代，一次 L1 缓存命中 1ns，一次内存访问 100ns——差了两个数量级" |

### Step 3：模式识别（对应 repo-story-time Phase 4）

从采集的信息中识别叙事元素：

| 元素 | 要识别的内容 |
|------|------------|
| 角色 | 谁做了什么决策？有没有"一个人 vs 所有人"的经典对抗？ |
| 主题 | 这个故事本质上是关于什么？性能 vs 安全？简单 vs 灵活？ |
| 冲突 | 真实的技术取舍点在哪里？有没有争论了很久的问题？ |
| 转折 | 什么东西改变了游戏规则？硬件突破？新理论？一个 bug？ |
| 悬念 | 有什么未解决的开放问题？读者可以参与思考什么？ |

### Step 4：叙事线编织

将采集的信息填入叙事模板：

```
选择叙事原型 → 背景（张力建立）→ 问题（具体痛苦）→ 演进（试错/突破/外部推力）
→ 方案（取舍解释 + 开放问题）
```

检查标准：
- [ ] 每个技术点都有自己的"位置"（不是为了写而写）
- [ ] 读者能感受到"为什么"（不是记住"是什么"）
- [ ] 有冲突/取舍/代价，而非完美的线性叙述
- [ ] 有具体年代、数据、人物、场景

## 5. 与现有方法论的关系

### 与问题驱动写作法（内容层）的关系

问题驱动法是**最小叙事单元**（每个知识点按"why→what→pain→verify"展开），叙事架构是**全局结构**（整篇文章的弧线）。

```
叙事线（全局）
  ├── 知识点 A → 问题驱动法展开
  ├── 知识点 B → 问题驱动法展开
  ├── 知识点 C → 问题驱动法展开
  └── 知识点 D → 问题驱动法展开
```

叙事架构决定**知识点的选择和排列顺序**，问题驱动法决定**每个知识点的展开方式**。

### 与七段式章节结构的关系

七段式是**结构骨架**，叙事线是**底层驱动力**：

| 七段 | 叙事视角 |
|------|---------|
| 段一（受众定位） | 融进"背景"中——"如果你做过 X，你一定遇到过 Y"本身就是背景叙事 |
| 段二（问题定义） | 问题 + 冲突的起点 |
| 段三-四（概念+原理） | 演进的展开或方案的深度拆解 |
| 段五（实践） | 演进的终点（"今天我们可以这样用"） |
| 段六（坑点） | 开放问题 + 取舍代价 |
| 段七（结尾） | 叙事收束（"所以这个技术给我们的启示是……"） |

### 与四层质检体系的关系

叙事架构在**逻辑层**和**风格层**之间新增了一个视角：

| 质检层 | 叙事相关检查项 |
|--------|--------------|
| 事实层 | 叙事中涉及的时间、人物、数据准确吗？ |
| 逻辑层 | 因果关系成立吗？演进动力有依据吗？冲突真实存在吗？ |
| **叙事层**（新增） | 有背景张力吗？问题是场景化的吗？读者感受到"为什么"了吗？有取舍/冲突/代价吗？ |
| 风格层 | 无 AI 套话、格式规范 |
| 传播层 | 标题有吸引力、开头有钩子 |

## 6. 示例对照

### 反面（知识点堆砌 + 只交付方案）

> MMU 是内存管理单元，负责将虚拟地址转换为物理地址。MMU 包含 TLB、页表遍历器、内存保护单元等组件。使用 MMU 可以实现虚拟内存、内存隔离、按需分页等功能。x86 架构使用四级页表，ARMv8-A 架构也使用类似机制。

### 正面（叙事线驱动）

> 1980 年代的计算机直接用物理地址跑程序。每开一个进程，都需要手动协商它在内存里的位置——装不下？重排。崩溃了？整个系统挂了。这就是当年"多任务"的实际成本。
>
> 设计师面对的取舍很明确：继续用物理地址，简单但无法隔离；引入地址转换，安全但引入延迟。MMU 就是这场取舍的产物——在每次内存访问中插入一层"翻译"，以 1-2 个周期的代价换取进程之间的完全隔离。
>
> 但翻译本身又成了瓶颈。为了解决这个问题，TLB 出现了——把最近用过的地址翻译结果缓存起来，命中率超过 99%。再到后来，页表从一级长成了四级（x86-64），TTW（页表遍历器）变成了硬件自动完成，每一层都是为了应对某个场景的瓶颈。
>
> 到这一步，核心矛盾不再是"要不要地址转换"，而是"转换开销怎么最小化"。四级页表的每一级，都在回答一个问题。

### 变化说明

| 维度 | 反面 | 正面 |
|------|------|------|
| 背景 | 无 | 1980 年代物理地址 + 多任务场景的价格 |
| 问题 | 无 | 无法隔离、需要手动协商 |
| 冲突 | 无 | "简单但无法隔离" vs "安全但引入延迟" |
| 演进 | 无 | TLB → 多级页表 → 硬件遍历器 |
| 取舍 | 无 | "以 1-2 个周期代价换取隔离" |
| 人物/场景 | 无 | "设计师面对的取舍""需要手动协商" |

## 7. 快速检查清单

完成写作后，逐项检查：

**背景层（Context）**：
- [ ] 有具体的年代/版本/硬件数据，不写"随着技术发展"
- [ ] 有当时的约束条件（硬件/成本/知识水平）
- [ ] 有导致的"痛苦"描述，而非抽象的问题陈述

**问题层（Problem）**：
- [ ] 问题放在具体场景里讲（"你在做 X 的时候……"）
- [ ] 问题有具体代价（不解决会怎样？损失多少？）
- [ ] 问题涉及真实取舍（不是"既 A 又 B"，而是"要 A 就得放弃 B"）

**演进层（Evolution）**：
- [ ] 每个阶段都有外部驱动因素（硬件变化/场景爆发/理论突破）
- [ ] 没有"事后诸葛亮"——早期方案在其时代是合理的
- [ ] 有试错、有争论、有转折点，而非平铺直叙

**方案层（Solution）**：
- [ ] 解释了取舍（为什么这样设计，不那样设计）
- [ ] 标注了局限性（还有什么没解决）
- [ ] 读者离开文章时能带走一个"设计哲学"层面的理解

## 与说服力的关系

叙事架构和说服力框架是**同一枚硬币的两面**：

| 叙事层 | 对应说服原则 | 作用 |
|--------|------------|------|
| 背景 | Unity + Social Proof | "你也经历过同样的处境"→建立共鸣 |
| 问题 | Scarcity + Authority | "大多数人不知道真实原因"→制造稀缺 |
| 演进 | Commitment | "从最初的方案到现在，一步步走来"→交付承诺 |
| 方案 | Authority + Reciprocity | "可验证的数据 + 可复现的实验"→ 给予价值 |

详细说服力框架见 `{skill_dir}/references/tech-writing-persuasion.md`。
