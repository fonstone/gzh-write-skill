# AI 刻板句式反模式识别

适配自 [stop-slop](https://github.com/hardikpandya/stop-slop) 的中文版 AI 写作模式检测清单。

## 使用方式

- 写作完成后逐层扫描，从一级到三级依次排查
- 每层全部通过后再进入下一层
- 命中任何模式 → 按对应改写方法修复

---

## 一级：短语层（禁用短语清单）

### 1.1 废话开场白（Throat-Clearing Openers）

直接说内容，不要预告你要说什么。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "这里的关键是……" | "这里的关键是，性能瓶颈不在 CPU 而在 IO。" | 直接说"性能瓶颈不在 CPU，在 IO。" |
| "问题是……" | "问题是，大多数团队都忽略了这一点。" | 直接陈述事实 |
| "说实话……"/"坦白说……" | "说实话，这个方案行不通。" | 去掉"说实话"，结论前置 |
| "值得注意的是……" | "值得注意的是，这个延迟波动在 5% 以内。" | "延迟波动在 5% 以内。" |
| "不得不说的是……" | "不得不说的是，这个设计有先天缺陷。" | "这个设计有先天缺陷——" |
| "我想说的是……" | "我想说的是，优化不是一步到位的。" | 去掉，直接说 |
| "有趣的是……" | "有趣的是，同样的代码在 ARM 上快了三倍。" | "同样的代码在 ARM 上快了三倍——原因在于……" |
| "你可能会问……" | "你可能会问，为什么要用 Rust？" | 直接回答，或删除设问 |
| "有没有想过……" | "你有没有想过，为什么 malloc 不直接返回物理地址？" | 直接说"malloc 不直接返回物理地址，因为……" |

**规则**：任何以"预告→内容"结构开头的句子 → 砍掉预告，只留内容。

### 1.2 强调拐杖（Emphasis Crutches）

没有任何实际含义，删掉。

| 禁用模式 | 示例 |
|---------|------|
| "请注意" | "请注意，这段代码不能在 ARM 上运行。" |
| "需要强调的是" | "需要强调的是，这不是一个通用的解决方案。" |
| "别误会" | "别误会，我并不是说 C 语言不好。" |
| "说真的" | "说真的，这个 bug 我查了两天。" |
| "划重点" | "划重点：页表遍历需要 4 次内存访问。" |
| "没错" | "没错，这就是为什么 eBPF 比内核模块安全。" |
| "这就意味着" | "这就意味着，每次上下文切换都要刷 TLB。" |

**规则**：去掉这些词之后读一遍——如果句子意思不变，删掉；如果句子没这些词就不成立，整个句子重写。

### 1.3 互联网黑话（替代 Business Jargon）

| 禁用词 | 替换为 |
|--------|--------|
| 赋能 | 让、帮助、支持 |
| 抓手 | 切入点、方法 |
| 闭环 | 完整流程、循环 |
| 对齐 | 一致、协调 |
| 链路 | 路径、流程 |
| 颗粒度 | 精细程度、详细程度 |
| 底层逻辑 | 根本原因、本质 |
| 顶层设计 | 总体规划、架构设计 |
| 落地 | 实现、部署、应用 |
| 痛点 | 具体问题、困难 |
| 方法论 | 方法、思路 |
| 矩阵 | 组合、体系 |
| 沉淀 | 积累、总结 |
| 复用 | 重复使用 |
| 击穿 | 突破、超越 |
| 心智 | 认知、理解 |
| 体感 | 感受、体验 |
| 透传 | 传递、传达 |
| 带宽（指精力） | 精力、时间 |
| 共建 | 合作、共同开发 |
| 延展 | 扩展、延伸 |
| 组合拳 | 多种方法结合 |

**规则**：全文搜索命中 ≥ 2 处则整体扣分。建议全书替换为白话。

### 1.4 副词与软化词（Adverbs & Softeners）

所有副词和软化词基本都可以删掉。

| 分类 | 禁用词 |
|------|--------|
| 程度强调 | 非常、极其、十分、相当、颇为、尤其 |
| 软化修饰 | 可能、或许、大概、似乎、有点、有些 |
| 时间虚词 | 近年来、长期以来、近来、目前 |
| 判断模糊 | 本质上、从根本上、从某种意义上说、某种程度上 |
| 冗余补位 | 进行、实现、做出、起到 |

**规则**：全文搜索，每处问"去掉这个词句子变弱了吗？"答案是否 → 删掉。

### 1.5 无信息增量套话（Vague Declaratives）

说重要但不说具体哪里重要。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "意义重大" | "这项技术的意义十分重大。" | 具体说对什么、在哪个方面产生了什么影响 |
| "影响深远" | "AI 对本行业的影响深远。" | 具体说哪个环节有什么变化 |
| "值得关注" | "这个趋势值得关注。" | 为什么值得关注？说出来 |
| "引发思考" | "这件事引发了我们的思考。" | 思考了什么？直接写出来 |
| "值得深思" | "这个问题值得深思。" | 深思考的是什么结论？ |

**规则**：如果一句话可以加上"说人话就是"然后接完全不同的话 → 原句是套话，重写。

### 1.6 元评论（Meta-Commentary）

不要评论文章的结构，让文章自己流动。

| 禁用模式 | 示例 |
|---------|------|
| "接下来我们讨论……" | "接下来我们讨论内核调度器的实现。" |
| "如前所述" | "如前所述，MMU 负责地址转换。" |
| "上文提到" | "上文提到过 TLB 刷新的问题。" |
| "我们将在后文展开" | "我们将在后文展开这个案例。" |
| "在本节中" | "在本节中，我们将分析三种方案。" |
| "这部分内容比较深入" | "这部分内容比较深入，初学者可以跳过。" |
| "写到这里我突然想到" | "写到这里我突然想到一个案例……" |

**规则**：删除所有对文章自身结构的评论。如果读者需要知道在文章的什么位置，加 H2/H3 标题就够了。

### 1.7 假亲密与夸张（Performative Emphasis）

| 禁用模式 | 示例 |
|---------|------|
| "我敢说" | "我敢说，90% 的开发者不知道这个。" |
| "我保证" | "读完这篇，我保证你能理解 MMU。" |
| "相信我" | "相信我，这个坑你一定会踩。" |
| "毫不夸张地说" | "毫不夸张地说，这是近十年最重要的技术。" |
| "懂的都懂" | "这个框架的坑，懂的都懂。" |
| "你品，你细品" | "这个设计的精妙之处，你品，你细品。" |

### 1.8 最高级与里程碑式拔高（Superlatives & Milestone Framing）

适配自 Humanizer-zh 的"绝对化强调、最高级和广泛影响"检测。LLM 倾向使用最高级表达拔高事物，制造虚假重要性。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "标志着" + 重大事件 | "标志着项目进入了一个全新阶段" | "项目启动了第二阶段" |
| "具有里程碑意义" | "这一版本的发布具有里程碑意义" | "这个版本增加了 X 和 Y 两个核心功能" |
| "奠定了……基础" | "为行业未来的发展奠定了坚实基础" | "让后续的 X 方案有了可行性" |
| "划时代的""开创性的""突破性的" | "这是一项划时代的突破" | 具体说突破了什么 |
| "作为……的证明/见证" | "作为公司创新能力的证明" | "公司的创新能力体现在——" |
| "具有深远影响" | "对行业格局产生了深远影响" | 具体说改变了哪个环节 |
| "重塑格局""改写规则""颠覆行业" | "将彻底重塑行业格局" | 具体说改变了什么 |
| "黄金时代""全新时代""新的纪元" | "标志着 AI 的黄金时代" | 去掉，只说具体变化 |

**规则**：任何以"标志/里程碑/证明/见证+程度拔高"结构的句子 → 替换为具体事实描述。问自己："去掉这些拔高词，这个事实本身成立吗？"

### 1.9 媒体引述与普遍认知（Media Citations & Consensus Framing）

适配自 Humanizer-zh 的"普遍认知度和媒体引述"检测。LLM 倾向引用"广泛认为""被广泛报道"来增强说服力，却很少提供具体来源。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "被广泛报道" | "这一事件被多家权威媒体广泛报道" | 指名道姓："《纽约时报》和路透社报道了" |
| "引发广泛关注/热议" | "该产品发布后引发了广泛关注" | 具体数据："发布 24 小时微博话题阅读量 2 亿" |
| "受到普遍关注" | "AI 安全问题受到了普遍关注" | "2024 年 3 月，欧盟通过《AI 法案》后——" |
| "专家指出/学者认为" (无具体来源) | "专家指出，这一趋势不可逆转" | 指名："斯坦福教授李飞飞在 NeurIPS 2024 上指出" |
| "研究表明/调查表明" (无出处) | "研究表明，用户的耐心正在下降" | "2024 年《Nature》子刊上的一篇论文发现——" |
| "业内人士分析" | "业内人士分析，市场将迎来洗牌" | "一位在芯片行业干了 15 年的朋友说——" |
| "众所周知/人人皆知" | "众所周知，数据是 AI 时代的石油" | 删掉，直接说事实 |

**规则**：所有"XX认为/报道/指出"必须有具体来源（人名+机构/媒体名称+时间）。找不到具体来源 → 删掉该引述。

### 1.10 "挑战与未来展望"固定框架（Challenge & Future Framework）

适配自 Humanizer-zh 的"文章结构式的挑战与未来展望框架"检测。LLM 死板遵循"简介-背景-现状-挑战-对策-结论"的议论文叙事框架。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "面临/遭遇/面对挑战" | "尽管取得了成就，但该行业也面临着独特的挑战" | 直接说具体问题，不用"挑战"框架 |
| "尽管面临这些挑战" + "展望未来" | "尽管面临这些挑战，未来前景依然乐观" | 删掉过渡，直接说现状和预期 |
| "任重道远/在路上" | "保护环境的道路依然任重道远" | 说具体还要做什么 |
| "未来发展方向/前进的道路" | "未来，公司的发展方向是——" | "公司计划明年——" |
| "放眼未来/放眼长远" | "放眼未来，AI 将在医疗领域发挥更大作用" | 删掉，说具体时间点的计划 |
| "挑战与机遇并存" | "在当前环境下，挑战与机遇并存" | 删掉，直接说是什么挑战和机遇 |
| "在竞争日益激烈的市场环境中" | | "2024 年市场增速放缓至 3%，竞争更加激烈" |

**规则**：所有"挑战→展望"过渡结构 → 砍掉框架性的"挑战"段落和"展望"段落，直接陈述具体问题和具体计划。真人的文章不一定非要有"挑战"这个段落。

### 1.11 协作/助手式语言（Collaborative Language）

适配自 Humanizer-zh 的"协作式或辅助性语言"检测。初稿残留了人机对话的语气。

| 禁用模式 | 示例 |
|---------|------|
| "当然可以" | "当然可以，我来帮你分析一下" |
| "请随时告诉我" | "如果需要任何调整，请随时告诉我" |
| "如有任何问题" | "如有任何问题，欢迎随时交流" |
| "很高兴为您" | "很高兴为您解答这个问题" |
| "如果您需要调整任何部分" | "如果您需要调整任何部分，请告诉我" |
| "让我逐一分析" | "让我逐一分析这几个方面" |
| "这个想法很有启发性" | "这是一个很有启发性的问题" |
| "希望这个回答对你有帮助" | 开头的礼貌性前缀 |

**规则**：全文搜索助手指征词。这种语言只在"人机对话"中出现，不会出现在正式文章中。发现一处删一处。

### 1.11 知识截止日期泄露（Knowledge Cutoff Leakage）

适配自 Humanizer-zh 的"知识截止日期和训练数据限制"检测。

| 禁用模式 | 示例 |
|---------|------|
| "截至 XX 年" | "截至 2025 年初，该技术尚未成熟" |
| "我的知识截止于" | "需要注意的是，我的训练数据截止于……" |
| "在公开资料中较为有限" | "虽然具体数据在公开渠道缺乏广泛记录" |
| "尚未有系统性记录" | "该领域的系统性研究尚不充分" |
| "缺乏广泛记录" | "该事件的详细资料在主流文献中缺乏记载" |

**规则**：这些是 LLM 训练数据限制的措辞，出现在文章中是明显的 AI 痕迹。直接删掉或替换为有具体来源的表述。

### 1.12 "从X到Y"抽象范围（Abstract Scope Framing）

适配自 Humanizer-zh 的"抽象范围"检测。LLM 倾向使用"从 X 到 Y"结构表示文中涵盖范围之广或跨越时间之长。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "从……到……" 涵盖范围 | "这本书涵盖了从历史起源到未来展望" | "这本书重点讲述了——"（选一个点切入） |
| "横跨……与……" | "横跨考古学、技术与哲学" | 只说最核心的两项 |
| "从远古到现代" | "从远古人类的迁徙到数字时代的 AI" | "本文聚焦于——"（缩小范围） |
| "贯穿……始终" | "这个矛盾贯穿了人类文明的始终" | "这个矛盾在云计算时代尤为突出" |

**规则**：全文搜索"从…到…"大跨度结构 → 砍到最具体的 1-2 个点。大范围=没焦点，没焦点=AI 味。

### 1.13 过度礼貌与过度谦虚（Excessive Politeness）

适配自 Humanizer-zh 的"过度礼貌和过度谦虚"检测。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "太棒了！您说得完全正确" | "太棒了！您提出的问题非常有深度" | 直接回答问题 |
| "让我茅塞顿开" | "您的洞见让我茅塞顿开" | 删掉 |
| "这是我读过的最有深度" | "这是我近期收到的最有价值的反馈" | 删掉 |
| "非常荣幸"/"深感荣幸" | "能为您服务深感荣幸" | 删掉 |
| "向您致以崇高的敬意" | | 删掉 |

**规则**：正式文章不需要对读者过度礼貌。正常表达即可。

### 1.13 "是"字判断句与"作为"结构过度使用（Copula Overuse）

适配自 Humanizer-zh 的"过度使用'是'字判断句和'作为'结构"检测。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "作为（身份/角色），X ……" | "作为一家领先的科技公司，X 公司致力于……" | "X 公司主要做——" |
| "是一项/是一个/是一种" (概括性定论) | "MMU 是一种用于地址转换的硬件单元" | "MMU（内存管理单元）负责地址转换" |
| "是……的关键/是……的基础" | "技术选型是整个项目的关键" | "技术选型决定了项目的开发效率和维护成本" |
| "是……的核心/是……的基石" | "数据是这个时代最核心的资产" | "数据的价值在于——" |

**规则**：全文搜索"作为"和"是"+"的关键/的基础/的核心" → 替换为具体描述。判断句让位于叙事句。

| 禁用模式 | 替换为 |
|--------|--------|
| 所有人 / 没有人 | 大多数 / 很少 |
| 总是 / 从不 | 经常 / 几乎不 |
| 永远 / 绝不 | 长期看 / 通常不 |
| 每个 / 所有 | 许多 / 大多数 |
| 完美 / 最优 | 足够好 / 适合大多数场景 |

**规则**：除非你能用数据证明绝对化表述（"100% 的用户"），否则改为条件化表达。

### 1.14 Emoji 滥用（Emoji Overuse）

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| Emoji 列表/标题装饰 | "📌 第一阶段：产品试点"、"✅ 关键成果：留存率提升" | 用纯文字，删除emoji |
| 大纲式 Emoji | "🔔 下一步：扩展市场"、"💡 核心建议" | 删掉 emoji |
| 段尾 Emoji | "读完这篇文章你会有答案 🚀"、"欢迎留言讨论 💬" | 删除 emoji |

**规则**：正式公众号文章禁止使用 Emoji 装饰标题和列表。如果在正文中出现偶发的单个 emoji（非装饰用途）且 ≤ 2 处，可容忍。

## 二级：句式层（结构性反模式）

### 2.1 二元对比结构（Binary Contrasts）

先否定 A 再说 B 的"不是 A，而是 B"句式。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "不是因为 X，而是因为 Y" | "性能问题不是因为 CPU 不够快，而是因为内存延迟。" | "性能瓶颈在内存延迟，不在 CPU。" |
| "X 不是问题，Y 才是" | "硬件不是问题，软件才是。" | 直接说"软件才是瓶颈。" |
| "不是 X，而是 Y" | "这不是一个技术问题，而是一个工程问题。" | "这是工程问题，不是技术问题。" |
| "看起来是 X，实际上是 Y" | "看起来是算法问题，实际上是数据问题。" | "问题的根因在数据，不在算法。" |
| "不是要 X，而是要 Y" | "不是要更快的硬件，而是要更聪明的算法。" | "需要更聪明的算法，而不是更快的硬件。" |

**规则**：全文搜索"不是"→检查是否构成二元对比 → 是则去掉否定部分，直接说肯定结论。

### 2.2 否定铺陈（Negative Listing）

先说不是什么再说是什么的"铺垫式"写法。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "不是 A……不是 B……而是 C" | "这不是一个简单的 bug。不是逻辑错误。不是边界条件。这是一个架构问题。" | "这是架构问题，不是逻辑bug或边界条件。" |
| "既不是 X，也不是 Y" | "这个问题既不是硬件引起的，也不是操作系统的问题。" | 直接说是谁的问题 |
| "无关 A 也无关 B" | "优化无关语言也无关框架，关键在于数据访问模式。" | "优化关键在于数据访问模式。" |

**规则**：列出 2 个以上否定的 → 砍掉全部否定，只留肯定句。

### 2.3 设问修辞（Rhetorical Setups）

自问自答的"问答模板"。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "为什么 X？因为 Y。" | "为什么要用 eBPF？因为它比内核模块安全。" | 直接说"eBPF 比内核模块安全，所以——" |
| "X 是什么？简单来说……" | "TLB 是什么？简单来说就是地址转换的缓存。" | "TLB 是地址转换的硬件缓存。" |
| "X 真的重要吗？答案是肯定的。" | "硬件选型真的重要吗？答案是肯定的。" | 直接说"硬件选型很重要，因为——" |
| "你可能想问……答案是……" | "你可能想问，这跟我有关系吗？答案是……" | 直接说关系是什么 |

**规则**：全文搜索问号 → 检查是否是设问 → 设问全部改为直接陈述。每篇文章最多保留 1 处设问（用于制造悬念时可以用一次）。

### 2.4 假拟人化（False Agency）

把无生命事物当作行为主体。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "数据告诉我们" | "数据告诉我们，eBPF 比传统方案快 3 倍。" | "我们测试发现，eBPF 比传统方案快 3 倍。" |
| "市场选择了" | "市场选择了 ARM 而不是 RISC-V。" | "大多数厂商选择了 ARM，因为——" |
| "技术推动了" | "云原生技术推动了微服务架构的普及。" | "微服务架构的普及得益于云原生技术。" |
| "趋势表明" | "趋势表明，Rust 正在替代 C 语言。" | "Rust 在系统编程领域的使用量在增长——" |
| "时间会证明" | "时间会证明 eBPF 的价值。" | "eBPF 的价值在于——" |
| "历史告诉我们" | "历史告诉我们，封闭生态最终会失败。" | "封闭生态最终会失败，因为——" |

**规则**：找到句子主语 → 如果主语不是人或组织 → 重写，把真正的人/团队/组织放回主语位置。找不到真人的就用"我们"。

### 2.5 同义词循环（Synonym Cycling）

适配自 Humanizer-zh 的"过度换词（同义词循环）"检测。AI 在描写同一人物或事物时，会刻意避免在相邻句子中重复同一名称，导致滥用同义词堆砌。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| 同一主体在相邻句中换不同称呼 | "这位艺术家……这位才华横溢的创作者……他……这位天才……" | 统一用"他"或固定称呼，不要反复换 |
| 同义替换堆砌 | "产品……该工具……这款解决方案……这个平台……" | "产品"从头叫到尾 |
| 抽象变体替换 | "公司……该机构……这个组织……企业……巨头" | "公司"就完了 |

**规则**：检查段落中同一实体的指代方式。如果在 3 句话内换了 3 种以上不同说法 → 统一为最直接的那个，去掉所有"诗意的变体"。真人写作不怕重复，AI 才怕重复。

**验证方法**：提取段落中所有对同一主体的称呼。超过 3 种不同的 → 同义词循环，需要统一。

### 2.6 旁白式叙事（Narrator-from-a-Distance）

站在高处俯瞰叙事，不把读者拉进场景。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "没有人想过……" | "没有人想过用 eBPF 来解决这个问题。" | "我们之前也没想过用 eBPF——直到——" |
| "人们往往会……" | "人们往往会在遇到性能问题时首先怀疑数据库。" | "我遇到性能问题时第一个怀疑的就是数据库——" |
| "开发者普遍认为……" | "开发者普遍认为 GC 语言比手动内存管理慢。" | "我刚学编程时也以为 GC 比手动管理慢——" |
| "在 X 场景下" | "在容器化部署场景下，网络延迟会成为瓶颈。" | "你容器化部署之后发现网络延迟上去了——" |
| "大多数人" | "大多数人低估了页表遍历的开销。" | "我之前也低估了页表遍历的开销，直到——" |

**规则**：用"你""我们""我"代替"人们""大家""开发者"。把读者拉进场景，不要站在外面当解说员。

### 2.7 翻译腔句式（直接保留自 writing-guide.md）

| 特征 | 示例 | 改写 |
|------|------|------|
| 主语充分化 | "读者需要理解这个概念的背景。" | "理解这个概念，需要先知道它的背景。" |
| 从句嵌套 | "我们之前讨论的那个在 ARM 平台上运行的方案……" | "我们在 ARM 上跑的那个方案——之前聊过——" |
| 被动语态 | "这个问题被认为是由内存泄漏导致的。" | "根因是内存泄漏。" |
| of 结构直译 | "对 CPU 调度器的优化" | "优化 CPU 调度器" |
| 长定语 | "这是一个在 Linux 5.15 以上版本中经过充分测试的内核模块。" | "这个内核模块在 Linux 5.15+ 上测试过。" |

**规则**：翻成英文再翻回中文 → 几乎一样 → 翻译腔，重写。

### 2.8 Wh- 开头的句子（中文适配）

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "什么是 X" | "什么是 MMU？MMU 是内存管理单元。" | "MMU（内存管理单元）负责——" |
| "为什么 X" | "为什么需要虚拟内存？因为——" | "虚拟内存的存在是因为——" |
| "如何 X" | "如何优化页表性能？第一步是——" | "优化页表性能的第一步是——" |
| "X 怎么样" | "eBPF 的性能怎么样？实测发现——" | "eBPF 的实测性能是——" |

**规则**：全文搜索"什么""为什么""如何""怎样"开头的句子 → 改为以结论/主语开头。

### 2.9 破碎短句堆叠（Dramatic Fragmentation）

连续使用短促的独立句制造"深刻感"。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "X。就是这样。Y。" | "调度器。就是这么简单。一个算法决定一切。" | "调度器本质就是一个算法决定一切。" |
| "X。接下来的 Y。" | "内存。接下来是文件系统。" | 直接用过渡句衔接 |
| "停顿。悬念。然后揭晓。" | "一次内存访问。4 次页表查询。这就是全部。" | "一次内存访问背后隐藏着 4 次页表查询。" |

**规则**：连续 2 个以上短独句段 → 合并成完整句子或调整节奏。

## 三级：节奏层（语感模式）

### 3.1 三连列表（Three-Item Lists）

AI 默认输出三项列举，这是最显著的模式指纹之一。

| 禁用模式 | 示例 | 改写 |
|---------|------|------|
| "X、Y 和 Z" | "MMU 有三大功能：地址转换、内存保护和缓存管理。" | 只用最重要的两项，或直接展开关键的一项 |
| "三个关键点" | "这里有三个关键点需要注意。" | 砍到一个或零个——让读者自己发现 |
| "一方面……另一方面……此外" | 三段式递进 | 改为两句或直接一段讲完 |

**规则**：全文搜索"三""三个""三大"等数字列举 → 改为两项或一项。三连是人类演讲的节奏，不是写作的节奏。

### 3.2 破折号滥用（Em-Dash Overuse）

| 禁用模式 | 示例 |
|---------|------|
| "X——Y" 连续出现 | 全文破折号超过 3 处 |
| 破折号代替逗号 | "这个方案——也是团队讨论最久的方案——最终被采纳了。" |

**规则**：全文破折号 ≤ 3 处。能用逗号或括号的不用破折号。

### 3.3 程式化段落结尾

每段结尾都用"一击必杀句"制造深刻感的模式。

| 禁用模式 | 示例 |
|---------|------|
| 段落结尾都是短句 | "这就是现实。""这才是本质。""仅此而已。" |
| 每段最后一句话加粗 | 全文 ≥ 3 处段尾加粗 |

**规则**：检查连续 3 段结尾结构 → 如果都是短句/加粗/总结式 → 至少改 2 段为平实收尾。

### 3.4 "So" 段首（中文适配）

| 禁用模式 | 示例 |
|---------|------|
| "所以，"段首 | "所以，MMU 是现代操作系统不可或缺的组件。" |
| "因此，"段首 | "因此，我们需要重新审视这个方案。" |
| "于是，"段首 | "于是，eBPF 诞生了。" |

**规则**：全文搜索段首的"所以""因此""于是"→ 至少砍掉一半，直接用内容衔接。

### 短语层
- [ ] 无废话开场白（"这里的关键是""值得注意的是"等）

- [ ] 无假亲密（"我敢说""相信我"等）
