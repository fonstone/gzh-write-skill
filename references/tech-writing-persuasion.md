# 技术写作说服力与注意力引导

方法源自 [CodingCossack/agent-skills-library](https://github.com/CodingCossack/agent-skills-library/tree/main/skills/writing-skills) 的 `persuasion-principles.md`（Cialdini 七原则 + Meincke et al. 2025 研究，N=28,000 LLM 对话验证）以及 `SKILL.md` 的信息架构方法论（分层结构、流程图决策、Token 效率、示例策略）。

该仓库本体是关于 AI Agent 技能文档的元方法论。以下是对其说服力原则和信息架构策略的技术写作适配。

---

## 1. 核心洞察

### 1.1 纯技术堆砌的问题根源

```
知识点 A → 知识点 B → 知识点 C → 知识点 D
没有"为什么现在要讲这个"，读者随时可以离开
```

**问题不在知识点密度，在缺乏"阅读动量"**——读者没有理由继续往下读。

### 1.2 说服力驱动的内容层

```
叙事线（背景→问题→演进→方案）   ← tech-narrative-architecture.md
  ↕ 每一层由说服原则驱动
说服力（承诺→证据→稀缺→验证）  ← 本文档
  ↕ 每个知识点由问题驱动展开
问题驱动法（why→what→pain→verify) ← tech-writing-guide.md 内容层
```

说服力是叙事线的**驱动力**：叙事线决定"讲什么"，说服力决定"为什么要继续读"。

## 2. 说服力七原则（技术写作适配）

### 2.1 权威（Authority）

**原则**：读者对专业背书、可验证数据、可追溯来源更信任。

| 技术写作应用 | 反例 | 正例 |
|------------|------|------|
| 数据来源标注 | "实测性能提升很大" | "在 Intel i7-12700 + CUDA 12.1 下测得，P50 延迟从 45ms 降到 12ms" |
| 引用标注 | "研究表明" | "根据 ACM SIGCOMM 2023 论文《……》第 4.2 节" |
| 环境声明 | "在我的机器上跑了" | "测试环境：Linux 6.1, gcc 12.2, -O2 编译, DDR5-4800" |
| 局限声明 | 无 | "但这个方法在 ARMv7 上没有硬件支持，适用场景有限" |

**硬规则**：一个技术结论如果没有可追溯的数据来源 → 整段可信度归零。

**黑名单**：
- ❌ "据调查" / "数据显示" / "研究表明" → 必须给出具体出处
- ❌ "在大多数场景下" → 必须给出具体场景范围
- ❌ "性能显著提升" → 必须给出具体数字 + 环境

### 2.2 承诺与一致（Commitment）

**原则**：读者一旦在标题/开头接受了你的承诺，会下意识地"追踪"承诺是否兑现。

#### 承诺链

```
标题承诺（"看完这篇，你能排查 90% 的 OOM"）
  ↓ 读者：好，我等着
开头重申（"这篇文章我会手把手带你走一遍排查路径"）
  ↓ 读者：确认了
正文兑现（每一步都交付给一个具体方法/命令/案例）
  ↓ 读者：得到了预期的东西
结尾回顾（"回到开头的问题——现在你知道从哪里查了"）
  ↓ 读者：承诺闭环，读完有满足感
```

#### 执行规则

| 原则 | 要求 |
|------|------|
| 标题承诺必须具体 | ❌ "深入理解内存管理" → ✅ "从 OOM 到根因：4 条命令定位内存泄漏" |
| 开头必须重申承诺 | 标题是钩子，开头是再度确认——第一段结尾必须有"读完本文，你将掌握……" |
| 正文必须逐条兑现 | 承诺了"4 条命令" → 4 条命令必须有编号、有输出、有解释 |
| 结尾必须回顾承诺 | "回到一开始的 OOM 场景——现在你应该知道从哪里开始排查了" |

**硬规则**：标题中的任何承诺如果在正文中没有对应段落 → 重写标题或补段落。

### 2.3 稀缺（Scarcity）

**原则**：读者对"有限的知识"比对"广泛的知识"更在意。

| 技术写作应用 | 反例 | 正例 |
|------------|------|------|
| 知识稀缺性 | "MMU 有三大功能" | "90% 的开发者以为 MMU 只做地址转换，但实际上它在——场景下的行为才是关键" |
| 版本稀缺 | "Linux 实现了" | "Linux 6.1 才合入这个特性，之前的版本用的是完全不同的机制" |
| 场景稀缺 | "在嵌入式场景中" | "在电池供电的 IoT 设备上，一次多余的页表遍历能多耗 0.5μJ——这个量级在手机上看不见，但在传感器节点上是致命的" |
| 认知缺口 | "了解这个很重要" | "你可能不知道，每次 malloc 后面至少有 3 次系统调用——这里才是性能真正的拐点" |

#### 稀缺制造公式

```
[常见认知] → 但 [一个大多数人不了解的具体事实/版本/场景/边界条件]
```

**不要**：
- 编造稀缺（"99% 的人都不知道" → 假的，读者不信任）
- 滥用稀缺（全文 5 处"你可能不知道" → 麻木了）

**要**：
- 用真实的版本差异、架构差异、场景差异打造稀缺
- 用"我踩过的坑"替代"你可能不知道"——更真诚

### 2.4 社会认同（Social Proof）

**原则**：读者更倾向于相信"大多数开发者已经验证过"的结论。

| 技术写作应用 | 反例 | 正例 |
|------------|------|------|
| 工程共识 | "这样做效率更高" | "Linux 内核社区在 5.15 版本之后统一采用了这个方案，因为它解决了之前 3 年都没解决的争用问题" |
| 常见错误 | "要注意边界条件" | "我们在 code review 中见过至少 20 次——新手都会忘检查返回值，结果线上出问题" |
| 行业趋势 | "这个方案越来越流行" | "2024 年的 kubecon 上，60% 的议题与此相关。不是炒作，是大家都在迁移" |

**执行规则**：
- 社会认同必须可验证（开源项目 star 数、会议占比、论文引用量）
- 用具体数字替代"越来越多""普遍认为"
- 每个"大多数"后面跟一个例外说明（"但在——场景下反而是少数"）

### 2.5 联盟感（Unity）

**原则**：读者对"我们是一类人"的写作者有更强信任。

| 技术写作应用 | 反例 | 正例 |
|------------|------|------|
| 共同身份 | "建议开发者关注" | "做嵌入式开发的同行，应该都经历过——" |
| 共同经历 | "这个 bug 是这样的" | "我记得第一次遇到这个 bug 时，debug 了一整个周末——最后发现是——" |
| 坦诚立场 | "这个方案有三个优点" | "这个方案我们团队用了半年，最后放弃在两个场景下——" |

**联盟感写作纪律**：
- 允许"我"（个人视角），但不允许"我"高高在上（"我来教你们"）
- "我们"的使用：排除性（"我们团队 vs 你们"→ 禁止）、包容性（"我们开发者"→ 允许）
- 坦诚局限比展示专业更有联盟感

### 2.6 互惠（Reciprocity）

**原则**：读者先获得价值，更倾向于回报（转发/收藏/关注）。

| 应用 | 执行 |
|------|------|
| 前置干货 | 文章前 30% 内容必须包含可立即使用的知识（命令/代码/排查方法） |
| 结尾行动 | "如果你遇到过类似问题，欢迎分享你的方案" |
| 互惠不强迫 | 不要在文章开头要求关注/转发 |

### 2.7 好感度（Liking）

**原则**：读者更愿意读完"喜欢"的人写的文章。

| 行为 | 效果 | 纪律 |
|------|------|------|
| 坦诚缺陷 | "这个优化在我们的项目里也失败过一次" | 只适用于 1-2 处，过多变成自我怀疑 |
| 适当幽默 | "是的，又是 mmap——它就像一个瑞士军刀，什么都能干但什么都不精" | 技术类幽默必须收敛，读者是来学知识的 |
| 避免居高临下 | "如果你还不理解" → "用我们团队的示例来说明" | 假设读者和你一样聪明，只是缺少特定场景经验 |

**好感度杀手（禁止）**：
- ❌ "这个问题其实很简单"（贬低读者）
- ❌ "稍微有点基础的人都知道"（贬低读者）
- ❌ "相信你已经掌握了"（虚假假设）
- ❌ "这部分内容比较深入"（制造距离感）

## 3. 段落说服节奏

### 3.1 RISE 段落节奏

借鉴 SKILL.md 的信息架构（Overview → When → How → Examples → Mistakes → Impact），适配为技术文章的段落节奏：

```
R — Recognize（认知）：先让读者意识到"这是个问题"
I — Identify（识别）：帮读者定位到具体场景/原因
S — Solve（解决）：给出解决方案
E — Evidence（证据）：用数据/代码/案例证明方案有效
```

**每个 RISE 循环 ≈ 1 个 H2 小节的内容量**。全文由多个 RISE 循环串联。

#### R：Recognize（认知建立）

**功能**：让读者产生"对对对，就是这样"的共鸣。

```markdown
# 写法
[场景] + [痛苦] + [你中过吗]

# 示例
你以为 malloc 返回 NULL 就安全了？不是。真正危险的是它返回了一个非 NULL 的指针，
但背后已经触发了一次 major page fault——你的实时线程在那几十微秒里已经错过了 deadline。
```

**说服力技巧**：Social Proof（"大多数嵌入式开发都经历过"）+ Unity（"我们团队也踩过"）

#### I：Identify（原因定位）

**功能**：解释问题背后的技术原因。

```markdown
# 写法
[表层表现] → [深层原因] → [机制解释]

# 示例
问题不在 malloc 本身，在 lazy allocation 的承诺：Linux 分配虚拟地址时不分配物理页，
真正分配是在第一次访问时（page fault）。这就是那把"延迟刀"。
```

**说服力技巧**：Authority（引用内核源码/官方文档）

#### S：Solve（方案交付）

**功能**：给出可操作的方法。

```markdown
# 写法
[解决方案] + [具体步骤] + [代码/命令]

# 示例
方案一：用 mlockall() 锁住进程的物理页面，强制在初始化阶段完成所有 page fault。
方案二：用 MAP_POPULATE 预分配……
```

**说服力技巧**：Commitment（兑现标题承诺）+ Scarcity（"大部分人不知道的 flag"）

#### E：Evidence（实证验证）

**功能**：用可验证的数据证明方案有效。

```markdown
# 写法
[测试代码] + [实测数据] + [环境标注]

# 示例
在 i7-12700 + Linux 6.1 上，2000 次 malloc+访问 耗时：
- 无 mlock：3.2ms（含 2000 次 page fault）
- 有 mlock：0.4μs（无 page fault）
```

**说服力技巧**：Authority（环境标注 + 可复现代码）

### 3.2 多循环串联

全篇文章 ≈ 3-5 个 RISE 循环。每个循环解决一个子问题，推进叙事线。

```
标题承诺（"从 OOM 到根因：4 条命令定位内存泄漏"）

RISE 循环 1：认识问题（为什么 OOM 不是瞬间发生的）
   → Recognize: "你遇到过服务半夜 OOM 重启吗？"
   → Identify: "OOM Killer 的触发条件是什么"
   → Solve: "用 dmesg 看 OOM 日志"
   → Evidence: 实际 OOM 日志 + 解读

RISE 循环 2：定位泄漏源
   → Recognize: "内存泄漏最常见的伪装是一点点涨上去"
   → Identify: "泄漏 vs 缓存占用的区别"
   → Solve: "用 /proc/meminfo + Smem 跟踪"
   → Evidence: 8 小时采样数据图

RISE 循环 3：根因分析
   → ...
```

**检查标准**：每个 RISE 循环都可以被**独立转载**——即放在任何平台上发表，读者也能完整理解一个技术点。

## 4. 读者注意力引导系统

### 4.1 三层注意力分配

| 层次 | 对应内容 | 目的是 |
|------|---------|--------|
| H2 标题 | 核心论点 | 扫读也能抓住文章主线 |
| 加粗/首句 | 每个段落的结论 | 跳读也能带走关键信息 |
| 正文 | 论证过程 + 数据 | 精读的完整交付 |

**执行规则**：
1. 一个 H2 只回答一个问题。如果回答了两个问题 → 分两个 H2
2. 每个段落的首句必须是该段的核心结论。首句不能是过渡（"此外""与此同时"等）
3. 正文中的关键数字、核心判断、可截图金句必须 **加粗**

### 4.2 认知资源曲线管理

读者在阅读过程中，认知资源呈倒 U 形分布：

```
【高】开头（最大关注）→ 标题承诺验证期
  ↓
【中】正文中段（稳定期）→ 需要每 3 分钟一个"刷新点"
  ↓
【低】结尾（开始滑走）→ 需要强收束
```

#### 开头（前 30%）：快速交付信任

| 规则 | 要求 |
|------|------|
| 300 字内建立"值得读"的证据 | 一个数据、一个案例、一个反常识 |
| 承诺清晰 | 读者读完第一段就知道这篇文章能给他什么 |
| 避免防御性写作 | "本文从多个角度分析了……" → 直接给结论 |

#### 正文中段（30%-80%）：注意力刷新点

每 3 分钟左右（≈ 7-10 个手机屏幕）设置一个刷新点：

| 刷新点类型 | 作用 | 使用频率 |
|-----------|------|---------|
| **知识点小结句** | "所以 X 的核心就是 Y"，让扫读读者带走结论 | 每个 H2 末 |
| **反面案例** | "大多数人会这样做——然后出问题" | 每 2-3 个 H2 |
| **节奏切换** | 从原理切换到类比，或从分析切换到实操 | 连续 3 段理论后强制切换 |
| **悬念钩子** | "但这里有一个隐藏的坑" | 每 3-4 个 H2 |

**刷新点的作用不是增加信息，是降低疲劳**。

#### 结尾（80%-100%）：承诺闭环

| 规则 | 要求 |
|------|------|
| 回顾标题承诺 | "回到文章开头的问题——现在可以回答了" |
| 一句话本质 | 全文的"所以这件事的本质是……" |
| 明确下一步 | "如果你想继续深入，可以看……" |

### 4.3 信息密度交替

从 content-enhance.md 继承，但用说服力原则增强：

```markdown
【高密度段】数据/论证/代码 → 权威性原则驱动
  读者：接受信息 → 信任建立

【低密度段】类比/感受/故事 → 联盟感原则驱动  
  读者：消化信息 → 情感共鸣

【悬念段】转折/未解问题 → 稀缺性原则驱动
  读者：产生了"然后呢"的期待
```

**硬规则**：不允许连续两段都是高密度（纯技术堆砌）或连续两段都是低密度（水分太多）。

## 5. 集成到现有方法论

### 与叙事架构的关系

```
tech-narrative-architecture.md（叙事线）
  ↓ 决定了：每段讲什么、按什么顺序
tech-writing-persuasion.md（说服力）
  ↓ 决定了：为什么要讲这段、读者为什么继续读
tech-writing-guide.md 内容层（问题驱动法）
  ↓ 决定了：每个知识点怎么展开
tech-writing-guide.md 行文层（节奏控制）
  ↓ 决定了：段落长度、密度交替、中途钩子
```

### 与四层质检体系的集成

新增说服层（位于逻辑层与风格层之间）：

| 质检层 | 说服相关检查项 |
|--------|--------------|
| 事实层 | 数据来源、测试环境、引用出处 — Authority |
| 逻辑层 | 承诺链是否完整（标题→开头→正文→结尾）— Commitment |
| **说服层**（新增） | 标题承诺是否在正文兑现？开头 300 字建立了"值得读"的证据？每个 RISE 循环可独立转载？认知资源曲线是否管理了？ |
| 风格层 | 无 AI 套话、格式规范、无居高临下语气 |
| 传播层 | 结尾有行动调用？读者读完有满足感？ |

## 6. 反面 → 正面示例

### Before（纯知识点堆砌）

> DMA 是 Direct Memory Access 的缩写。它允许外设直接读写系统内存而不需要 CPU 参与。DMA 有三种传输模式：单次传输、块传输和请求传输。使用 DMA 可以大大提高数据传输效率。

### After（RISE + 说服力驱动）

> 你在写一个高速数据采集程序——ADC 每秒产生 10 万次采样，每次中断通知 CPU。你的 RTOS 在中断里已经花了 60% 的时间，根本没有余量做其他任务了。
>
> **大多数嵌入式开发者的第一反应是优化中断处理程序**，但问题不在处理速度，在中断本身——每次传输 CPU 都要停下来保存上下文、跳转、恢复。这是 CPU 在"迁就"外设。
>
> DMA 的思路是把"谁迁移谁"反过来：**让外设自己去内存写数据，CPU 只在一批数据准备好了之后收到一个中断**。这个模式在 Linux 内核中由 `dmaengine` 框架实现。
>
> **实测对比（STM32H743 + AD7606, 100ksps, 16bit）**：
> | 模式 | CPU 占用 | 数据吞吐 | 可并行任务数 |
> |------|---------|---------|------------|
> | 中断模式 | 62% | 100% | 1（只够跑采集） |
> | DMA 模式 | 8% | 100% | 4（采集+FFT+显示+通信） |
>
> **所以 DMA 的本质不是"让传输变快"，是"把 CPU 从传输中解放出来"**。设计者的取舍很明确：用多一点硬件逻辑，换取 CPU 时间。

### 变化说明

| 维度 | Before | After |
|------|--------|-------|
| 开头 | 定义开场（无引力） | 具体场景 + 痛苦（Recognize + Unity） |
| 承诺 | 无 | "下面会告诉你根本原因"（Commitment） |
| 概念解释 | 翻译三个模式 | 展示设计思路转变（Identify） |
| 证据 | 无 | 实测对比表 + 硬件标注（Authority） |
| 结论 | "提高效率" | "解放 CPU"的本质陈述 + 取舍（收束） |
| 段落节奏 | 三段纯知识点 | 场景→分析→方案→证据→本质（RISE 完整循环） |
