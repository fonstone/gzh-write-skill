# 技术科普写作框架

本文档是 frameworks.md 的补充。在原有 7 套通用框架基础上，增加 4 套技术科普专用框架。

## 框架选择指南

| 文章类型 | 推荐框架 | 适用场景 |
|---------|---------|---------|
| 讲一个技术/概念的演进历程 | 原理演进型 | OS 调度器演进、文件系统发展史、AI 模型架构演变 |
| 两个相似技术的深度对比 | 技术对比型 | RISC vs CISC、eBPF vs 内核模块、Transformer vs Mamba |
| 手把手技术实现 | 实践指南型 | 编译 Linux 内核、部署 k8s 集群、实现一个简易 RTOS |
| 关键算法/内核源码分析 | 源码拆解型 | Linux 调度器源码分析、malloc 实现、page fault 处理路径 |
| **覆盖多层技术的全景分析** | **全景架构型** | AI Infra 全栈架构、云原生可观测性体系、GPU 集群方案对比 |

## 框架一：原理演进型

### 叙事驱动的演进（必读优先）

所有框架必须先理解叙事架构（`{skill_dir}/references/tech-narrative-architecture.md`），每个框架对应一种叙事原型。

| 框架 | 推荐叙事原型 | 核心驱动力 |
|------|------------|-----------|
| 原理演进型 | 演进史式 | "旧方案撑不住了 → 试错 → 突破" |
| 技术对比型 | 对抗式 | "主流观点是 X → 但有人发现 X 错了 → 范式转移" |
| 实践指南型 | 揭秘式 | "表面看起来是 X → 背后是 Y → 设计意图是 Z" |
| 源码拆解型 | 揭秘式 | "表面看起来是 X → 背后是 Y → 设计意图是 Z" |
| **全景架构型** | **地图式** | "你的系统卡在哪 → 逐层排查 → 找到瓶颈层 → 用真实案例验证诊断" |

### 结构（叙事化版本）

```
叙事背景（不是"历史背景"，而是"当时的张力"——什么矛盾在积累）
  ↓
冲突起点（旧方案在某场景下不可接受地失效了）
  ↓
第一次突破（新方案解决了当前问题，制造了下一阶段的新问题）
  ↓
演进动力（具体外因——硬件/场景/需求变化）
  ↓
现代方案（今天的取舍结果）
  ↓
开放问题（还有什么没解决）
```
| 段 | 占比 | 字数 |
|----|------|------|
| 背景 | 15% | ≈400 |
| 原始方案 | 20% | ≈500 |
| 演进动力 | 20% | ≈500 |
| 现代方案 | 35% | ≈900 |
| 哲学总结 | 10% | ≈200 |

### 注意事项
- 避免"事后诸葛亮"写法（"以前的方案太蠢了"），要说明当时为什么合理
- 每个阶段的代码/数据/架构图最好都要有
- 演进动力必须结合具体硬件/场景/需求变化，不要笼统的"随着技术发展"

## 框架二：技术对比型

### 结构（叙事化版本）

```
叙事背景（"当时所有人都默认用 X——因为 Y 还不够好"）
  ↓
挑战者出现（"但有一个团队/场景中发现，X 在某些条件下并不成立"）
  ↓
对抗展开（X 的优势在哪里，Y 的优势在哪里——你输我赢不是主题，各自有取舍才是）
  ↓
对比表（多维度：性能/复杂度/生态/适用场景——每个维度都对应一个具体取舍）
  ↓
选型建议（"什么场景选 A，什么场景选 B"——不是"看情况"，而是给明确的分界条件）
  ↓
开放问题（"这场对抗还在继续吗？下一代方案是什么？"）
```

### 对比表示例
| 维度 | eBPF | 内核模块 |
|------|------|---------|
| 安全性 | ✅ 验证器 + 沙箱执行 | ❌ 无保护，可崩溃内核 |
| 性能 | ⚠️ 有 JIT 编译开销 | ✅ 原生代码运行 |
| 事件源 | 内核插桩 + tracepoint + perf | 任意内核函数 |
| 开发复杂度 | 低（C/ Rust 子集） | 高（完整内核编程） |
| 适用版本 | Linux 4.4+ | 所有版本 |

### 注意事项
- 对比表是核心产出，必须覆盖所有关键维度
- 选型建议不能"都要"或"看情况"——给出明确的分界条件
- 如果两方有优劣争议，要标注各自的最佳适用场景

## 框架三：实践指南型

### 结构（叙事化版本）

```
序幕（"你打开了一个生产环境 issue，报错信息只有一行——OOM"——用具体场景拉开序幕）
  ↓
问题定义（"你要解决的问题是：在物理内存耗尽之前，怎么预测和干预？"）
  ↓
前置准备（工具、权限、环境）
  ↓
分步实现（每一步都回答"这一步为什么是必要的，不做什么会怎样"）
  ↓
验证方法（"你怎么知道你做对了？——看 /proc/meminfo 的这行"）
  ↓
踩坑记录（"实际生产中，你在 X 场景会遇到 Z 情况，因为……"——每个踩坑都关联一个具体原因）
  ↓
收束（"回到开头那个 issue——现在你知道从哪里查了"）
```

### 每步格式
```markdown
### Step N: 步骤名称

**目标**：这一步要做什么

```bash
# 环境依赖（如果有）
```

**操作**：
1. ___
2. ___
3. ___

**预期结果**：执行成功后你会看到什么
```

### 注意事项
- 每一步必须包含：操作 + 预期结果
- 踩坑记录必须有真实案例，不要编
- 验证方法要具体：不看日志看什么命令？输出什么算成功？

## 框架四：源码拆解型

### 结构（叙事化版本）

```
叙事钩子（"Linux 内核处理一个 page fault 只需要几十微秒——这几十微秒里发生了什么？"）
  ↓
机制概述（"page fault 的本质是 CPU 在虚拟地址和物理地址之间找不到映射——为什么会有这个缺口？"——每个解释都要回答"为什么"）
  ↓
核心数据结构（"要理解 page fault 处理路径，先看懂三个 struct——它们记录了谁在等什么、缺了什么、从哪补"——每个数据结构对应一个"被什么问题催生"）
  ↓
关键路径代码（"去掉 error handling 后，主线代码只有 15 行——每一行都是一个决策"——逐行标注"这一行回答了什么取舍"）
  ↓
运行流程（"用 ftrace 跟着跑一遍，看每一级页表遍历背后的逻辑"）
  ↓
复杂度分析（"最坏 O(n)，但为什么实际接近 O(1)？因为 MMU 的硬件预读和 TLB"）
  ↓
设计意图还原（"所以 Linux 的 page fault 处理路径不是一堆 if-else，而是对'缺页的所有可能性'逐级降级处理——每一级都比上一级代价更大"）
```

### 代码格式
```c
// 为可读性做了精简：去掉了错误处理、锁、调试日志
// 完整代码见: mm/memory.c:3210-3250
static vm_fault_t handle_pte_fault(struct vm_fault *vmf)
{
    pte_t entry;
    
    if (!vmf->pte)        // PTE 不存在
        return do_anonymous_page(vmf);
    
    if (!pte_present(vmf->orig_pte))  // 页面被换出
        return do_swap_page(vmf);
    
    // ...
}
```

### 注意事项
- 删减代码必须明确标注删了什么、为什么删
- 关键路径要和数据结构对应上
- 复杂度分析要有理有据：
  ❌ "时间复杂度是 O(n)"
  ✅ "最坏情况需要遍历长度为 n 的链表，所以是 O(n)；但实际中由于 __，平均接近 O(1)"

## 框架五：全景架构型（处理多层技术的全景主题）

### 设计动机

当主题涉及多个技术层（如 AI Infra 全栈、云原生可观测性体系、GPU 集群方案），单一技术的深度展开不足以给出全局判断，但平均分配到每层又会变成"什么都没讲透"。

全景架构型的核心策略是：**广度梳理 → 深度选打 → 案例闭环**。

### 叙事原型：地图式

| 框架 | 推荐叙事原型 | 核心驱动力 |
|------|------------|-----------|
| 全景架构型 | 地图式 | "你的系统卡在哪 → 逐层排查 → 找到瓶颈层 → 用真实案例验证诊断" |

地图式的叙事逻辑不是"从 A 到 Z 全部讲一遍"，而是：

```
定位（"你在哪一层？"）→ 扫描（每层一张快照）→ 锁定（找到瓶颈层）→ 深挖（进入瓶颈层内部）→ 验证（案例证明诊断正确）
```

### 结构

```
全景定位（"大多数人的 AI 系统卡在中间两层，而不是两端"——给出全局判断，建立"地图"）
  ↓
逐层扫描（每层只回答 3 个问题：核心矛盾 + 典型误区 + 选型框架，每层配一张架构快照图）
  ↓ ↓ ↓
瓶颈锁定（基于扫描，指出哪一层是当前最常见的瓶颈，给出判断依据）
  ↓
深度拆解瓶颈层（进入该层内部，用原理 + 代码/配置/数据展开，必须包含可操作的具体方案）
  ↓
案例闭环（"我们帮一个团队做了三个改动……"——具体场景、具体改动、具体数据）
  ↓
延伸判断（对其他层的影响、层间依赖关系、"先修哪层"的优先级建议）
```

### 七段式骨架映射

| 段 | 功能 | 叙事角色 | 占比 | 字数 | 纪律 |
|----|------|---------|------|------|------|
| 一 | 全景定位 | 建立地图 | 10% | ≈200-300 | 用一个反常识或全局判断开头（"不是 X 的问题，是 Y"），不给逐层列表 |
| 二 | 逐层扫描 | 地图遍历 | 25% | ≈500-800 | 每层一张架构快照图（ASCII 流程图/对比表），每层≤200 字，只说核心矛盾 |
| 三 | 瓶颈锁定 | 诊断结论 | 10% | ≈200-300 | 给出"为什么这层是当前最常见瓶颈"的判断依据 |
| 四 | 深度拆解瓶颈层 | 深挖 | 30% | ≈600-1000 | 原理展开 + 代码/配置/命令 + 架构图，禁止纯理论 |
| 五 | 案例闭环 | 验证 | 20% | ≈400-600 | **强制**：具体场景 + 具体改动 + 具体数据（见 tech-writing-guide 真实案例闭环章节） |
| 六 | 层间依赖与优先级 | 延伸判断 | 5% | ≈100-200 | 回答"先修哪层"、"层间如何相互影响" |
| 七 | 结尾 | 收束与 CTA | 5% | ≈100-150 | 使用 ending_templates.md 模板 |

### 纪律

**与七段式骨架的关键差异**：
- **字数不是硬约束，深度是**：段四、段五没有字数上限——如果瓶颈层需要 2000 字才能讲透，就用 2000 字。段二如果 400 字能扫完 8 层，就用 400 字。
- **段二（逐层扫描）必须配架构快照图**：每层至少 1 张 ASCII 流程图 / 对比表 / 层级关系图。禁止纯文字描述层间关系。
- **段五（案例闭环）必须包含三要素**：具体场景（公司/团队/问题）、具体改动（before→after）、具体数据（延迟/吞吐/准确率绝对值）。
- **层间依赖优先于层内细节**：如果两层之间的依赖关系是核心见解（如"推理延迟高不一定是推理框架的问题，通常是存储 IO"），优先展开层间关系。

### 架构图/流程图规范

全景架构型文章在每个关键节点必须包含可视化元素：

| 位置 | 必需的可视化类型 | 示例 |
|------|----------------|------|
| 段二开头 | 全局架构层级图（ASCII/对比表） | 8 层 AI Infra 堆栈图 |
| 每层扫描 | 单层架构快照（流程图/框图） | 推理层的 Prefill→Decode→Cache 流程 |
| 段三瓶颈锁定 | 瓶颈定位图（热力图/对比表） | 各层延迟占比热力图 |
| 段四深度拆解 | 原理流程图/架构图 | KV Cache 管理流程 / 并行策略选型树 |
| 段五案例闭环 | before/after 对比表或架构变化图 | 改动前后系统架构对比 |

### 与内容增强策略的配合

全景架构型使用以下增强策略组合：

| 增强策略 | 在全景架构型中的用途 |
|---------|-------------------|
| **角度发现** | 找到"大多数人的认知盲区"，确定段一的开篇判断 |
| **密度强化** | 在瓶颈层（段四）补充具体参数/配置/命令 |
| **细节锚定** | 在案例闭环（段五）提供可复现的测试条件和数据 |
| **真实体感** | 案例必须是真实团队的实际踩坑经历，禁止编造 |
