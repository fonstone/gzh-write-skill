# 技术科普写作框架

本文档是 frameworks.md 的补充。在原有 7 套通用框架基础上，增加 4 套技术科普专用框架。

## 框架选择指南

| 文章类型 | 推荐框架 | 适用场景 |
|---------|---------|---------|
| 讲一个技术/概念的演进历程 | 原理演进型 | OS 调度器演进、文件系统发展史、AI 模型架构演变 |
| 两个相似技术的深度对比 | 技术对比型 | RISC vs CISC、eBPF vs 内核模块、Transformer vs Mamba |
| 手把手技术实现 | 实践指南型 | 编译 Linux 内核、部署 k8s 集群、实现一个简易 RTOS |
| 关键算法/内核源码分析 | 源码拆解型 | Linux 调度器源码分析、malloc 实现、page fault 处理路径 |

---

## 框架一：原理演进型

### 结构
```
历史背景（当时的问题是什么）
  ↓
原始方案及局限（最初是怎么解决的，有什么问题）
  ↓
演进动力（是什么推动了这个变化——硬件/场景/需求）
  ↓
现代方案（现在的设计思路，核心机制）
  ↓
设计哲学总结（"所以这个演进告诉我们的道理是…"）
```

### 字数分配
| 段 | 占比 | 字数 |
|----|------|------|
| 背景 | 15% | ≈400 |
| 原始方案 | 20% | ≈500 |
| 演进动力 | 20% | ≈500 |
| 现代方案 | 35% | ≈900 |
| 哲学总结 | 10% | ≈200 |

### 注意事项
- 避免"事后诸葛亮"写法（"以前的方案太蠢了"），要说明当时为什么合理
- 每个阶段的代码/数据/架构图最好都要有
- 演进动力必须结合具体硬件/场景/需求变化，不要笼统的"随着技术发展"

---

## 框架二：技术对比型

### 结构
```
共同目标（两个方案要解决什么问题）
  ↓
方案 A 核心原理（怎么做）+ 优势 + 局限
  ↓
方案 B 核心原理（怎么做）+ 优势 + 局限
  ↓
多维度对比表（性能/复杂度/生态/适用场景）
  ↓
选型建议（什么场景选 A，什么场景选 B）
```

### 对比表示例
| 维度 | eBPF | 内核模块 |
|------|------|---------|
| 安全性 | ✅ 验证器 + 沙箱执行 | ❌ 无保护，可崩溃内核 |
| 性能 | ⚠️ 有 JIT 编译开销 | ✅ 原生代码运行 |
| 事件源 | 内核插桩 + tracepoint + perf | 任意内核函数 |
| 开发复杂度 | 低（C/ Rust 子集） | 高（完整内核编程） |
| 适用版本 | Linux 4.4+ | 所有版本 |

### 注意事项
- 对比表是核心产出，必须覆盖所有关键维度
- 选型建议不能"都要"或"看情况"——给出明确的分界条件
- 如果两方有优劣争议，要标注各自的最佳适用场景

## 框架三：实践指南型

### 结构
```
问题定义（要解决什么具体问题）
  ↓
前置准备（硬件/软件/权限要求）
  ↓
分步实现（Step 1 / Step 2 / Step 3 …）
  ↓
验证方法（怎么知道你做对了）
  ↓
踩坑记录（常见的失败模式和解决方法）
```

### 每步格式
```markdown
### Step N: 步骤名称

**目标**：这一步要做什么

```bash
# 环境依赖（如果有）
```

**操作**：
1. ___
2. ___
3. ___

**预期结果**：执行成功后你会看到什么
```

### 注意事项
- 每一步必须包含：操作 + 预期结果
- 踩坑记录必须有真实案例，不要编
- 验证方法要具体：不看日志看什么命令？输出什么算成功？

---

## 框架四：源码拆解型

### 结构
```
机制概述（这个机制是做什么的，为什么需要它）
  ↓
核心数据结构（关键 struct / 变量 / 宏）
  ↓
关键路径代码（主线代码流程，去掉异常处理分支）
  ↓
运行流程（一步步 debug / trace 跟踪）
  ↓
复杂度分析（时间/空间复杂度 + 为什么这样设计）
```

### 代码格式
```c
// 为可读性做了精简：去掉了错误处理、锁、调试日志
// 完整代码见: mm/memory.c:3210-3250
static vm_fault_t handle_pte_fault(struct vm_fault *vmf)
{
    pte_t entry;
    
    if (!vmf->pte)        // PTE 不存在
        return do_anonymous_page(vmf);
    
    if (!pte_present(vmf->orig_pte))  // 页面被换出
        return do_swap_page(vmf);
    
    // ...
}
```

### 注意事项
- 删减代码必须明确标注删了什么、为什么删
- 关键路径要和数据结构对应上
- 复杂度分析要有理有据：
  ❌ "时间复杂度是 O(n)"
  ✅ "最坏情况需要遍历长度为 n 的链表，所以是 O(n)；但实际中由于 __，平均接近 O(1)"
